Hard Rules to Remember

  • MAX 4 parallel developers - More task groups are split into execution phases
  • QA Expert only runs when integration/E2E tests exist AND testing mode = full
  • Orchestrator coordinates only - It spawns agents, never implements code
  • PM alone sends BAZINGA - Only after Validator confirms success
  • Security tasks force SSE - Auth/crypto/security tasks route to Senior Software Engineer

1. Configuration Deep Dive

BAZINGA's behavior is controlled by three JSON configuration files in the bazinga/ folder:

File Purpose
model_selection.json Which AI model each agent uses (haiku/sonnet/opus)
skills_config.json Which skills are available/mandatory for each agent
challenge_levels.json QA Expert's 5-level test challenge configuration

1.1 Model Selection Strategy

Each agent runs on a specific model. The default assignments balance cost and capability:

Agent Default Model Rationale
Developer Haiku Cost-efficient for straightforward implementation
Senior Software Engineer Sonnet Handles escalated complexity and failures
QA Expert Sonnet Balanced for test generation and validation
Tech Lead Opus Always Opus - critical architectural decisions
Project Manager Opus Always Opus - strategic planning, final quality gate
Configuration Scenarios
  • "I want faster iterations" → Keep Developer on Haiku
  • "I need bulletproof code" → Upgrade Developer to Sonnet
  • "Cost is no concern" → Set all agents to Opus

1.2 Testing Configuration

Configure testing mode using the slash command:

Claude Code
/bazinga.configure-testing

Three modes are available:

Mode QA Expert Tests Required Use Case
Full Enabled Unit + Integration + E2E Production code, quality-critical
Minimal Disabled Unit tests only Rapid iteration, development
Disabled Disabled Lint checks only Prototyping (not recommended for production)

QA Expert's 5-Level Challenge System

When QA Expert runs (mode=full with tests), it applies progressive testing challenges:

Level Name What It Tests On Failure
1 Boundary Probing Edge cases, null values, max/min Developer retry
2 Mutation Analysis Code mutations to verify test completeness Developer retry
3 Behavioral Contracts Pre/post conditions, invariants Escalate to SSE
4 Security Adversary Injection attacks, auth bypass Escalate to SSE
5 Production Chaos Network failures, race conditions Escalate to SSE

1.3 Skills Configuration

Skills are specialized capabilities agents can invoke. Configure them with:

Claude Code
/bazinga.configure-skills

Two preset profiles:

Profile Skills Included Best For
Lite (default) lint-check, security-scan, test-coverage Fast development, standard projects
Advanced All skills including codebase-analysis, pattern-miner, api-contract-validation Complex projects, comprehensive analysis

Key Skills

Skill What It Does Used By
security-scan Detects vulnerabilities (SQL injection, XSS, hardcoded secrets) Tech Lead (mandatory)
test-coverage Generates coverage reports, identifies untested code Tech Lead (mandatory)
lint-check Code quality, style, best practices Developer, Tech Lead
codebase-analysis Finds similar features, reusable utilities SSE (mandatory)

2. Parallel Mode Mastery

2.1 Understanding Execution Modes

The PM decides between two execution modes based on your requirements:

Simple Mode
  • 1 Developer at a time
  • Sequential execution
  • Best for: Single focused tasks
  • Faster per-task, but serial
Parallel Mode
  • 2-4 Developers simultaneously
  • Concurrent execution
  • Best for: Multi-component features
  • Faster overall for complex work
Hard Limit: MAX 4 Parallel Developers

If PM identifies more than 4 independent task groups, they're split into execution phases:

  • Phase 1: Groups 1-4 (parallel)
  • Phase 2: Groups 5-8 (parallel, after Phase 1 completes)
  • And so on...

2.2 Structuring Requests for Parallelism

PM looks for independent work that can run concurrently:

Good for Parallel
Build dashboard with 4 widgets:
- User stats widget
- Revenue chart widget
- Activity feed widget
- Notifications widget

4 independent components = 4 parallel developers

Sequential Dependencies
Build auth system:
1. First create User model
2. Then add auth middleware
3. Then create login endpoint
4. Finally add session management

Each step depends on previous = simple mode

2.3 Monitoring Parallel Execution

During parallel execution, you'll see status for each group:

Parallel Mode Transcript
PM: Created 4 task groups. Mode: PARALLEL
- US1: User stats widget
- US2: Revenue chart widget
- US3: Activity feed widget
- US4: Notifications widget
ORCHESTRATOR: Spawning 4 developers in parallel...
DEV[US1]: User stats widget complete. READY_FOR_REVIEW
DEV[US3]: Activity feed widget complete. READY_FOR_REVIEW
DEV[US2]: Revenue chart widget complete. READY_FOR_REVIEW
DEV[US4]: Notifications widget complete. READY_FOR_REVIEW
ORCHESTRATOR: All groups complete. Routing to Tech Lead...

Each group flows through the pipeline independently (Dev → QA → Tech Lead), but they run concurrently.

3. Advanced Orchestration Commands

3.1 Standard Orchestration

Claude Code
/bazinga.orchestrate [requirements]

Use for: Clear, well-defined tasks where you know what you want.

3.2 Advanced Orchestration (with Discovery)

Claude Code
/bazinga.orchestrate-advanced [requirements]

Adds a Requirements Engineer phase before PM:

  1. Requirements Engineer clarifies ambiguities
  2. Analyzes codebase for reusable components
  3. Identifies complexity and parallelization opportunities
  4. Generates Enhanced Requirements Document
  5. Then PM receives enriched context
When to Use Advanced
  • Vague or ambiguous requirements
  • Large unfamiliar codebase
  • Complex refactoring
  • Production-critical features

3.3 Spec-Kit Orchestration

Claude Code
/bazinga.orchestrate-from-spec

For planning-first workflows. Expects spec-kit files in .specify/features/:

  • spec.md - Feature specification
  • tasks.md - Task breakdown with markers
  • plan.md - Implementation plan

Task Markers

Marker Meaning
[P] Parallelizable task
[US1] User story group 1
[R1] Research task (pre-implementation)

3.4 Choosing the Right Command

Situation Command
Clear requirements, know what you want /bazinga.orchestrate
Vague requirements, need discovery /bazinga.orchestrate-advanced
Already have spec-kit files /bazinga.orchestrate-from-spec

4. Escalation & Quality Gates

4.1 Escalation Triggers

BAZINGA automatically escalates when issues arise:

Trigger Action
Developer fails 1 time Escalate to SSE (Sonnet)
QA Level 3+ fails Escalate to SSE (Sonnet)
Security-sensitive task detected Force SSE + mandatory Tech Lead
Complex blocker Spawn Investigator (Opus)
Tech Lead blocked 3+ times Escalate to Opus model
Security-Sensitive Tasks

Tasks containing keywords like auth, security, crypto, password, or token automatically route to Senior Software Engineer with mandatory Tech Lead review, regardless of complexity.

4.2 Quality Gates

Every piece of code passes through multiple quality checkpoints:

Developer Gate

Unit tests + lint checks must pass before proceeding.

QA Gate (if enabled)

Integration/E2E tests with 5-level challenge progression.

Tech Lead Gate

Security scan + test coverage + code review. Must approve or request changes.

PM Gate

Verifies all success criteria before attempting BAZINGA.

Validator Gate

Independent verification before accepting BAZINGA.

5. The Validation Protocol

This section covers the BAZINGA validation protocol - the final quality gate that ensures "BAZINGA" truly means "done."

5.1 Success Criteria

During planning, PM creates success criteria stored in the database:

Example Success Criteria
[
  "Rate limiter middleware exists at src/middleware/rateLimiter.ts",
  "Unit tests cover >80% of limiter logic",
  "Integration test verifies 429 response after limit exceeded",
  "Security scan passes with no high/medium findings",
  "Code review approved by Tech Lead"
]

5.2 The BAZINGA Flow

PM Checks Criteria

PM queries the database to verify each success criterion is met.

PM Sends BAZINGA

If all criteria pass, PM sends BAZINGA to the Orchestrator.

Orchestrator Spawns Validator

The Orchestrator spawns an independent Validator agent.

Validator Verifies Independently

Validator runs tests, checks criteria, and verifies claims without trusting PM's word.

Accept or Reject

ACCEPT: Project complete!
REJECT: PM re-spawned with findings to fix issues.

Zero Tolerance Policy

The Validator has zero tolerance for incomplete work:

  • One failing test = REJECT
  • One unmet criterion = REJECT
  • Security issue found = REJECT

This ensures BAZINGA is a reliable signal of true completion.

5.3 Rejection Handling

When Validator rejects, the flow continues:

Rejection & Recovery
PM: All criteria verified. BAZINGA!
ORCHESTRATOR: Spawning Validator for independent verification...
VALIDATOR: REJECT
- Integration test failing: expected 429, got 200
- Coverage at 72%, required 80%
ORCHESTRATOR: Re-spawning PM with validator findings...
PM: Addressing validator findings. Spawning Developer to fix...

6. Troubleshooting Guide

Common Issues

Problem Likely Cause Solution
Orchestration seems stuck Agent processing complex task Wait longer; check logs in bazinga/bazinga.db
Developer keeps failing Task too complex for Haiku Let escalation happen, or upgrade Developer to Sonnet
QA keeps rejecting Tests too strict or code quality issues Review QA feedback; consider minimal testing mode
BAZINGA rejected multiple times Systematic issue with implementation Review validator findings; may need to restart with clearer requirements
No QA phase running Testing mode not "full" or no tests written Check /bazinga.configure-testing; ensure Dev writes tests

Reading Orchestration Logs

All orchestration state is stored in bazinga/bazinga.db (SQLite). Key tables:

  • sessions - Orchestration sessions with status
  • orchestration_logs - Timestamped agent interactions
  • task_groups - All task groups and their status
  • success_criteria - PM's success criteria for the project

When to Restart vs Continue

Continue (let it retry)
  • Normal failures and retries
  • Escalation happening as expected
  • Progress being made each iteration
Restart with clearer requirements
  • Same error repeating 3+ times
  • PM asking clarifying questions
  • Fundamental misunderstanding of task
