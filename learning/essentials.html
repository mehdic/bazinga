Hard Rules to Remember

  • MAX 4 parallel developers - More task groups are split into phases
  • QA Expert only runs when integration/E2E tests exist AND testing mode = full
  • Orchestrator coordinates only - It spawns agents, never implements code
  • PM alone sends BAZINGA - Only after validator confirms success
  • Security tasks force SSE - Auth/crypto work routes to Senior Software Engineer + Tech Lead

1. Welcome to Your AI Dev Team

1.1 What is BAZINGA?

BAZINGA is a multi-agent orchestration system for Claude Code. Instead of one AI handling everything, specialized agents collaborate like a real development team:

PM → Developer → QA Expert → Tech Lead → PM → BAZINGA!

Meet the Team

Agent Role Key Responsibilities
Project Manager Strategic planning & completion Analyzes requirements, creates task groups, decides mode (simple/parallel), sends BAZINGA when done
Developer Implementation Writes code, creates tests, reports status (READY_FOR_QA or READY_FOR_REVIEW)
Senior Software Engineer Escalated complexity Handles failures, security-sensitive code, Level 3+ challenge failures
QA Expert Quality assurance Runs integration/E2E tests, applies 5-level challenge progression
Tech Lead Code review & security Reviews architecture, runs security scans, approves or requests changes
Investigator Deep analysis Root cause analysis for complex blockers

1.2 Why Multi-Agent vs Single-Agent?

When should you use BAZINGA instead of regular Claude Code?

Use BAZINGA for:
  • Complex features with multiple components
  • Quality-critical code needing review
  • Tasks requiring multiple skills (dev + test + review)
  • Large refactoring efforts
  • Security-sensitive implementations
Use Solo Claude for:
  • Quick fixes and small edits
  • Exploratory coding / prototyping
  • Simple questions about code
  • File operations and searches
  • Learning and experimentation
Key Insight

BAZINGA adds overhead (multiple agents, reviews, validation) that pays off for complex tasks but is unnecessary for quick fixes. Choose the right tool for the job.

2. Your First Orchestration

2.1 Running an Orchestration

Start an orchestration with the slash command:

Claude Code
/bazinga.orchestrate [your requirements here]

Example:

Claude Code
/bazinga.orchestrate Add a rate limiter middleware that limits requests to 100/minute per IP address, with configurable limits and Redis-backed storage

2.2 Understanding the Agent Flow

Here's what happens during a typical orchestration:

Phase 1: Planning

The Orchestrator spawns the Project Manager, who analyzes your requirements, creates task groups, and decides whether to use simple mode (1 developer) or parallel mode (2-4 developers).

Phase 2: Implementation

Developer(s) implement the code and write tests. They report either READY_FOR_QA (tests written) or READY_FOR_REVIEW (no tests).

Phase 3: Quality Assurance

If tests exist and mode is "full", QA Expert runs integration and E2E tests using a 5-level challenge system. Results: PASS or FAIL.

Phase 4: Code Review

Tech Lead reviews code quality, runs security scans, and checks test coverage. Results: APPROVED or CHANGES_REQUESTED.

Phase 5: Completion

Project Manager verifies all success criteria are met, then sends BAZINGA! to signal completion.

2.3 The Routing Logic

From Status Routes To
Developer READY_FOR_QA QA Expert (if tests exist & mode=full)
Developer READY_FOR_REVIEW Tech Lead (directly)
QA Expert PASS Tech Lead
QA Expert FAIL (Levels 1-2) Developer (retry)
QA Expert FAIL (Levels 3-5) Senior Software Engineer (escalate)
Tech Lead APPROVED Project Manager
Tech Lead CHANGES_REQUESTED Developer (fix & retry)

3. Anatomy of a Good Request

3.1 Crafting Effective Prompts

The quality of your request directly affects the quality of the output. Be specific about what you want.

Too Vague
/bazinga.orchestrate Fix the bug

PM will ask clarifying questions, wasting time.

Specific & Clear
/bazinga.orchestrate Fix the login timeout bug in auth.js that causes sessions to expire after 30 seconds instead of 30 minutes

Clear problem, location, expected behavior.

3.2 Good Request Components

Component Example Why It Helps
What "Add rate limiting middleware" Core functionality requested
Where "in the Express app at src/middleware/" Location context
How "using Redis for distributed storage" Technical constraints
Acceptance "should return 429 when limit exceeded" Success criteria

3.3 Request Examples by Type

Feature Request

Claude Code
/bazinga.orchestrate Add user profile avatars:
- Allow uploading images (max 5MB, png/jpg/gif)
- Store in S3 with presigned URLs
- Add avatar field to User model
- Create upload/delete API endpoints
- Include image resizing to 200x200

Bug Fix

Claude Code
/bazinga.orchestrate Fix the race condition in OrderService.processPayment() where concurrent requests can double-charge customers. The bug occurs when two payment webhooks arrive within 100ms of each other.

Refactoring

Claude Code
/bazinga.orchestrate Refactor the monolithic UserController into separate controllers:
- AuthController (login, logout, register)
- ProfileController (get, update, delete)
- PreferencesController (settings, notifications)
Maintain existing API contracts and add tests for each.

3.4 Common Beginner Mistakes

Avoid These Mistakes
  • Too vague - "Fix the issues" gives PM nothing to work with
  • Too micro-managing - "First create file X, then add line Y at position Z" undermines agent autonomy
  • Interrupting - Stopping orchestration mid-flow loses context; let it finish
  • No acceptance criteria - How does PM know when it's done?

4. Interpreting Results

4.1 Success Indicators

A successful orchestration ends with BAZINGA! from the PM:

Success Transcript
PM: All task groups completed. Success criteria verified:
- Rate limiter middleware created
- Tests passing (12/12)
- Security scan clean
- Code review approved

BAZINGA!

4.2 Understanding Failures & Retries

Seeing failures and retries is normal behavior. BAZINGA is designed to handle them:

Normal Retry Flow
DEVELOPER: Implementation complete. Status: READY_FOR_QA
QA: Level 2 (Mutation Analysis) - FAIL
Mutation at line 45 not caught by tests.
DEVELOPER: Added test for edge case. Status: READY_FOR_QA
QA: All levels passed. Status: PASS
Retries Are Good

When you see Developer retry after QA failure, the system is working as designed. Each iteration improves code quality.

4.3 Escalation Chain

When issues persist, BAZINGA escalates to stronger models:

Trigger Escalation
Developer fails 1 time Escalate to Senior Software Engineer
QA Level 3+ fails Escalate to Senior Software Engineer
Tech Lead blocked 3+ times Escalate to Opus model
Complex blocker Spawn Investigator for root cause analysis

4.4 The Validation Layer

Before BAZINGA is accepted, an independent Validator verifies:

  • All success criteria are actually met
  • Tests pass independently
  • No critical issues were missed

If the Validator rejects, PM is re-spawned with the findings to address issues.

Zero Tolerance Policy

The Validator has zero tolerance for incomplete work. One failing test = rejection. This ensures BAZINGA truly means "done."
